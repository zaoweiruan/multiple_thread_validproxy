#### 1. 修复 LOG_ERROR 解析 bug，补全所有命令模式日志级别设置 (2026-05-06)
- **File**: `memory/project_knowledge.md` (lines 339-362)
- **Module**: Logger / main.cpp
- **Problem Type**: bug (runtime_error)
- **Relevance**: Directly addresses the LOG_ERROR parse bug and log level application to all command modes
- **Key Insight**: 
  - **LOG_ERROR 解析 bug**: `Logger::stringToLevel()` 将 `"LOG_ERROR"` 转为小写 `"log_error"`，但匹配的是 `"error"`，导致 `config.json` 中 `"file_level": "LOG_ERROR"` 被错误解析为 `LogLevel::INFO`
  - **修复方法**: 添加对 `"log_"` 前缀的处理（如 `"log_error"` → `"error"`），同时兼容带和不带 `LOG_` 前缀的格式
  - **日志级别未在所有模式应用**: 以下命令模式未调用 `setFileLevel()` 和 `setConsoleLevel()`：
    - `generator`、`show-sub`、`find-proxy`/`findminproxy`、`sync`、`dedup`、`import-sub` (文件分支)、`default` 模式
  - **修复**: 在 `Logger::init()` + `setFileEnabled()` 后添加 `setFileLevel()` 和 `setConsoleLevel()` 调用
  - **config.json 格式问题**: `"file_level": "LOG_ERROR"` 应改为 `"ERROR"` 以匹配枚举命名
- **Severity**: high

#### 5. Logger 日志重构需求与实现 (2026-04-22)
- **File**: `docs/日志重构需求.md` (full document)
- **Module**: Logger, main.cpp, ConfigReader
- **Problem Type**: design_pattern (logger_refactoring)
- **Relevance**: Documents the logger refactoring architecture, including mutex usage for thread safety
- **Key Insight**:
  - **日志等级**: TRACE(0) → DEBUG(1) → INFO(2) → WARN(3) → LOG_ERROR(4)
  - **分层输出**: console_level 控制台等级，file_level 文件等级
  - **配置项**: `config.json` 中新增 log.level/log.console_level/log.file_level
  - **线程安全**: 使用 mutex 保证多线程安全
  - **ERROR 是 Windows 宏**: 代码中使用 LOG_ERROR 而非 ERROR
  - **Logger 类实现** (from `docs/plans/2026-04-13-module-refactoring-plan.md`):
    - 使用 `std::mutex mutex_` 静态成员
    - 所有写操作使用 `std::lock_guard<std::mutex> lock(mutex_)`
    - `init()`, `write()`, `writeTimestamp()`, `flush()`, `close()` 都加锁
  - **潜在风险**: 日志等级字符串解析需要处理大小写和前缀（LOG_ vs log_）
- **Severity**: medium

### Recommendations

1. **LOG_ERROR 解析 bug**: 
   - Verify that `Logger::stringToLevel()` correctly handles both `"LOG_ERROR"` and `"ERROR"` formats
   - The fix should strip `"log_"` prefix (case-insensitive) before matching

2. **Log levels for all command modes**:
   - Ensure ALL 10 command modes have `setFileLevel()` and `setConsoleLevel()` calls after `Logger::init()` and `setFileEnabled()`
   - Modes to check: `generator`, `show-sub`, `find-proxy`, `findminproxy`, `sync`, `dedup`, `import-sub` (file), `import-sub` (URL), `default`, `tourl`, `-T`/`-U`/`-UA`

3. **Mutex locks in Logger**:
   - Verify all Logger methods that access shared resources (`outFile_`, `logDir_`, etc.) use `std::lock_guard<std::mutex>`
   - Methods to check: `init()`, `write()`, `writeTimestamp()`, `flush()`, `close()`, `setFileLevel()`, `setConsoleLevel()`, `setFileEnabled()`

4. **ConfigReader type/value validation**:
   - Add validation for all parsed values (e.g., `timeout_ms > 0`, `workers > 0`, `start_port` is valid port)
   - Consider adding validation helper functions like `validatePositiveInt()`, `validatePort()`, `validateUrl()`
   - Follow the placeholder pattern from the `blacklist_threshold` fix for configurable values

5. **Resource leaks in main.cpp**:
   - Check all `new` allocations have corresponding `delete` (or use smart pointers)
   - Verify `Logger::close()` is called before program exit
   - Check `XrayManager::release()` is called to clean up xray instances
   - Review all `sqlite3_*` resource usage: `sqlite3_close()`, `sqlite3_finalize()` for prepared statements
   - Look for any `FILE*` or `std::ofstream*` that may not be properly closed

6. **Past patterns to follow**:
   - Use `std::lock_guard<std::mutex>` for all mutex locks (RAII pattern)
   - Remove debug output (`std::cerr`, `fprintf(stderr, ...)`) in favor of `Logger::write()`
   - Apply config validation after parsing, before using values
   - Clean up deprecated/unused fields proactively
